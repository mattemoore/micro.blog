I am not a super technical leader. How do I know that my team is on the right track?

Jul 21, 2022

Even if you are not the most technical leader you can point your team in the right direction.

Which of these two environments does your team operate in?

martinfowler.com/articles/… (slightly reworded)

  1. team comes into work in the morning and checks the applciation monitoring dashboard…knows based on data that the application is running green…this means they can work on fun stuff instead of fixing whatever broke last night
  2. team can pull down the latest code, make an incremental code change that is quickly validated via unit tests, end-to-end tests, load tests, accessibility tests, security tests etc.
  3. team can focus can work on value-add items uninterrupted for most of the day
  4. team goes home happy, excited to come back in the morning add more value to the product

martinfowler.com/articles/… (slightly reworded)

  1. team comes into work and begins the day immediately dealing with a number of alerts for problems in production
  2. team is forced to check multiple logging and monitoring systems mixed with their assumptions to find the source of error as there are no aggregated logs across systems
  3. team works with PM during (misused) standup and assumes that the alerts are false positives
  4. team cannot work uninterrupted for a few hours as the day has many status meetings
  5. team goes home unhappy as they were not able to improve the product today and fear what is awaiting them when they log in tomorrow morning

Operatability is something that must be built into v1

Jul 14, 2022

Launching a new system is great and should be celebrated but this is also when new concerns start.

If you do not know the answers to the following questions before you launch you are at risk:

  1. Do we know exactly how far our system can scale?
  • How many users can the system handle at the same time before weird behaviours start to occur?
  • How many records can we have in the database before weird behaviours start to occur?
  1. Do we know what these weird behaviours are in advance so we can separate scaling issues from bugs?
  2. Will we know that weird behaviours have started before our users tell us?
  3. Do we have an operations dashboard where we can look at usage metrics, logs and run adhoc diagnostics so we can trace weird behaviours?
  4. In summary, will we be able to determe root causes of operational issues using data or will we be using our assumptions?

If you do not have these before your launch make sure you get these done before v2 and make sure you have them in place before you launch v1 of your next system.

PS. If you do not have #4 you can not validate the assumptions you made while building v1 and thus cannot know if you built the wrong thing.

Competency is the most important. Team fit comes next.

Jul 14, 2022

If you are a manager and cannot honestly determine the competency of a developer on your own i.e not get snowed, ask a developer you know for sure to be competent and have that developer review a small piece of the candidate’s code while you listen in.

This enables you to:

  1. Confirm the competency of the candidate
  2. Confirm the candidate can play nice with others when challenged

  1. “When you click this button it will show a diff of changes that you can look at before submitting the draft for approval.”
